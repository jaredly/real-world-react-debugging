Linting PropTypes ðŸ”§ðŸ› 

  • react/prop-types ðŸ”§
  • react/forbid-prop-types ðŸ› 
Prop Types can be a really valuable pattern for catching bugs -- React will warn you if a client is using a component in an unexpected way. They also serve as great documentation -- when looking at a component, I can read the prop types & often get a good idea of what's expected. However, they do take some maintainance, and it's not too hard for them to get out of sync with your actual component. That's where the `react/prop-types` rule comes in -- it will error if you use a prop in your component that you didn't specify in the propTypes declaration. We use the `forbid-prop-types` rule to raise an error if you use the catch-all prop types "array & object", which are too general to be useful documentation or to catch many bugs. However, sometimes you just want to get things done and not worry about prop types until later -- and these lint rules will get in the way. So be aware of the tradeoffs.

Automated Testing

  • Fixtures ðŸ› 

@jaredforsyth

Now, I've historically been very skeptical of UI testing. I've kept an eye on UI testing frameworks that come up, and there are several cool ones specifically for react, but always I'm reminded of an experience I had at one of the first places I worked. It was a very large team, gnarly codebase with lots of shared state & stuff, and there was this huge suite of UI tests that took like an hour to run, and had a whole team of 6 engineers whose sole job was to maintain the tests. And that's not what I want. For anythign short of a massive organization, this would be overkill, and even then I'd think twice. So my general take as been "if you've got any testable logic going on, don't couple it to your UI -- extract it into a pure function (or class) and test it like normal." However, that doesn't end up being good enough, especially if you're in a large organization. And at Khan Academy, we have 1 and a half thousand components -- we want some more garentees that they're all doing reasonable things. So in addition to aggressive linting (there are a ton more rules than the ones I just mentioned), we use fixtures.

Automated Testing

  • Fixtures ðŸ› 
  • Jest Snapshot tests ðŸ”§
  • Visual regression tests ðŸ› 

@jaredforsyth

So that was fixtures. However, the fixture testing that we do currently at KA only verifies that they don't literally throw an error in the render method. There are two tools that I've been eying that will take things a step further to ensure that you're not unknowingly breaking things, but that also don't look like they will add too much friction to the development process. These are Jest Snapshot tests & Visual regression tests. Jest snapshot tests store a JSON representation of the HTML output of your components -- so not the higher-level components, just the div, span, etc that end up being rendered to the page. Later, after making a change, you can regenerate this "snapshot" and compare it to the previous one. That way, you can see the impact that your changes had on the component at a DOM level, to make sure that your expectations for the change line up with reality. I think there's definitely work to be done to make this a seamless process though; there are plenty of ways to set this up such that it is a pain. But the key thing is -- you don't write the expected output by hand. The tools do that for you, notify you of any differences, and ask if you want to overwrite the current snapshot, or if you want to fix your code. Visual regression tests can function in much the same way. You fire up a browser with your rendered component, take a screenshot, and compare that to the screenshot you have stored on file. An added bonus to this is that you now have a record of what all of your components look like, so that a new developer coming in can get a much better idea of what things look like & how they work.
